## P1 — Confusing / likely support tickets

### 1. ~~`pt-18` is not a valid Tailwind class~~ **INVALID — false positive**
**Resolution (2026-07-21):** The project uses **Tailwind v4** (`tailwindcss ^4.1.17`), which has a dynamic spacing scale: any numeric suffix multiplies by `--spacing` (default `0.25rem`). `pt-18` evaluates to `18 × 0.25rem = 72px` and is fully valid. Verified at runtime: `getComputedStyle(main).paddingTop` returns `72px` on the no-banner path. Original audit incorrectly applied Tailwind v3 rules (fixed scale jumping 16 → 20). **No fix needed; no test needed.**

### 2. ~~"rent for free (self-rental)" and "Rent Free" button~~ **FIXED — ambiguous copy on dashboard**
**Resolution (2026-07-21, commit `c528d3da`):** Subtitle renamed to *"Your own offerings — provision a test instance on demand (self-rental)"* and button relabelled to **Test Provision** with `title="Provision this offering to yourself at no charge"`. RED→GREEN test in `dashboard-overview.spec.ts` asserts the new wording and absence of "rent for free".
**Route + lines:** `/dashboard/+page.svelte:787-788` (heading) and `:854-859` (button)
**Severity:** P1 (canonical ambiguous-copy example, cf. issue #433)
**Fix size:** small

**Fix sketch:** Rename the section to something like *"Your Offerings — quick self-provisioning"* and the button to *"Test Provision"* with a tooltip explaining it spawns an instance against the provider's own offering at no platform commission. Add a one-line explainer under the heading.

### 3. Offerings page has no `AuthRequiredCard` — shows red error box instead
**Route + lines:** `/dashboard/offerings/+page.svelte:63-66` (error assignment) and `:671-676` (rendering)
**Severity:** P1
**Fix size:** small

Sister pages (`rentals` line 487, `invoices` line 120, `account` line 105) all render `<AuthRequiredCard>` for unauthenticated visitors with a "Login / Create Account" button. The offerings page instead sets `error = 'Please authenticate to view your offerings'` and renders it inside a red `Error loading offerings` panel. An anonymous visitor who lands here (e.g., via deep-link or after session timeout) sees a hard error with no recovery path.

**User impact:** Deep-link a provider to `/dashboard/offerings` after their session expires; expects a sign-in prompt; gets a red "Error loading offerings: Please authenticate to view your offerings" with no login button.
**Fix sketch:** Gate the page on `isAuthenticated` like rentals/invoices do and render `<AuthRequiredCard subtext="Create an account or login to view and manage your offerings." />`. The page already imports `authStore`; add a subscribe in `onMount` (mirroring `rentals/+page.svelte:321`).

### 4. Click-to-cycle visibility / stock buttons surprise the user
**Route + lines:** `/dashboard/offerings/+page.svelte:224-237` (handlers) and `:777-794` (buttons)
**Severity:** P1
**Fix size:** large

The visibility button silently cycles `public → shared → private → public` on each click; the stock button cycles `in_stock → out_of_stock → discontinued`. The only affordance is a `title` tooltip ("Click to cycle visibility: public → shared → private") which (a) is invisible on touch devices, (b) tells the user the *order* but not what the *next* click will do, and (c) doesn't convey that flipping to "Private" hides the offering from the public marketplace. A user who meant to inspect the "Public" badge and clicked it accidentally just hid their offering.

**User impact:** Provider clicks "Public" badge expecting details or a toggle; it silently flips to "Shared"; offering may stop appearing on the marketplace; provider sees fewer requests with no explanation.
**Fix sketch:** Replace the cycle button with a small dropdown menu (`<details>` or a `Popover`) listing all three states with the current one marked, plus a one-line description per option ("Private — hidden from public marketplace"). File as an issue; touches the offerings card layout and likely warrants a shared `VisibilityMenu` component.

### 5. Invoice table provider column shows raw pubkey hex
**Route + lines:** `/dashboard/invoices/+page.svelte:217-219`
**Severity:** P1
**Fix size:** small (medium if backend lookup needed)

```svelte
<td class="px-4 py-3 text-neutral-400 font-mono text-xs">
  {truncateHash(contract.provider_pubkey)}
</td>
```

The "Provider" column shows a truncated hex pubkey (`0xabc123…`) instead of a provider name or `@username`. By contrast, the `rentals/+page.svelte:909-918` provider cell at least links to `/dashboard/reputation/{pubkey}` so the user can resolve the name with one click — but the invoice table doesn't even link. Users who want to find all invoices from a specific provider must remember 8-char hex prefixes.

**User impact:** Tenant downloads three invoices from three different providers; opens them later; expects to see provider names in the table; sees three rows of `0x1a2b3c…` and can't tell which is which without clicking into each contract.
**Fix sketch:** Wrap the pubkey in an `<a href="/dashboard/reputation/{pubkey}">` (mirrors rentals) and/or resolve the `@username` via the existing provider lookup pattern. Even just making it a link is a one-line small fix.

### 6. Account page stuck on "Loading..." if account fetch silently fails
**Route + lines:** `/dashboard/account/+page.svelte:104-106, 266-268`
**Severity:** P1
**Fix size:** small

The main render branches are:
```
{#if !isAuthenticated}  <AuthRequiredCard />
{:else if currentIdentity?.account}  ... full page ...
{:else}  <p>Loading...</p>
{/if}
```

There is no error path. If `authStore` reports `isAuthenticated=true` but `currentIdentity.account` is null (e.g., the `/accounts/{pubkey}` fetch failed silently, account record was soft-deleted, or the backend returned an empty 200), the user sits on a permanent "Loading..." with no recovery.

**User impact:** User signs in successfully but the account fetch fails (network blip, edge-case data); expects their settings page; gets "Loading..." forever, no error, no retry, no logout CTA.
**Fix sketch:** Distinguish loading from missing-account: track an `accountLoadFailed` state from the auth subscription and render an explicit error card with "Retry" and "Logout" buttons in the `{:else}` branch.

### 7. Email-verification banner preempts seed-phrase backup banner (account-recovery risk)
**Route + lines:** `/dashboard/+layout.svelte:58-63, 97-103`
**Severity:** P1
**Fix size:** medium

The banner switch is mutually exclusive:
```svelte
showEmailVerificationBanner = isAuthenticated && account && !account.emailVerified && !isCheckoutOrMarketplacePage
showSeedPhraseBackupBanner  = isAuthenticated && !showEmailVerificationBanner && activeIdentity?.type === 'seedPhrase' && !seedBackupDismissed
```

A seed-phrase user with an unverified email will **never** see the seed-phrase backup warning until they verify their email. Since the seed phrase is the only account-recovery mechanism for that identity type, hiding the backup nudge behind an unrelated gate is a real account-loss risk.

**User impact:** Seed-phrase user signs up, doesn't verify email, never sees "back up your seed phrase" prompt; loses browser storage; cannot recover account; support cannot help.
**Fix sketch:** Stack the two banners rather than switching between them (each dismissable independently). Adjust the `pt-*` padding math in `<main>` to account for stacked banners. Worth a separate design pass — file as issue.

### 8. Marketplace sort UI is desktop-only — mobile users cannot sort
**Route + lines:** `/dashboard/marketplace/+page.svelte:1182-1196`
**Severity:** P1
**Fix size:** small-medium

The "Price ↑ / Price ↓ / Reliability ↓" sort pills are wrapped in `hidden md:flex`. Mobile users see "{N} offerings found" but have no way to influence sort order — the only sort affordance on mobile is the quick-filter pills above (`Recently Added`, `Most Trusted`), which conflate sort with filter.

**User impact:** Mobile user wants the cheapest GPU; expects to sort by price ascending; cannot; scrolls through 100 unsorted offerings.
**Fix sketch:** Add a `<select>` sort dropdown visible on mobile (and on desktop as an a11y-friendly alternative to the pills). Wire to the same `sortField`/`sortDir` state. Quick fix, but verify it integrates with `syncFiltersToUrl`.

### 9. "Recipes only" filter has no explanation — canonical ambiguous copy
**Route + lines:** `/dashboard/marketplace/+page.svelte:1080-1083` (and the table headers at `:1453, :1754` use "recipe" / "View recipe script")
**Severity:** P1
**Fix size:** small

The advanced filters panel has three adjacent checkboxes: `Show demo offerings`, `Show offline offerings`, `Recipes only`. A new user has no way to know that "recipe" means `post_provision_script` (a setup script the provider baked into the offering). The expanded-row description calls the same thing a "recipe script" — also unexplained.

**User impact:** User wants a turnkey offering; sees "Recipes only" filter; doesn't know what it does; either ignores it or toggles it without understanding the result.
**Fix sketch:** Rename to "Includes setup script" or "1-click provisioning" and add a short tooltip/help text describing what the script does. Update the expanded-row label to match. Single-file copy change.

### 10. ~~"Gateway routing is being configured"~~ **FIXED — no ETA, no in-section refresh**
**Resolution (2026-07-21, commit `9b15d162`):** Pending-gateway copy now reads *"Gateway routing is being configured. Connection details will appear here shortly (typically 1–3 minutes)."* with an inline **Refresh** button scoped to the card that calls `location.reload()`. RED→GREEN test in `rentals.spec.ts` seeds a contract with `provisioning_instance_details` + NULL gateway fields and asserts the ETA copy + Refresh button.
**Route + lines:** `/dashboard/rentals/+page.svelte:979-981`
**Severity:** P1
**Fix size:** small

When a contract has `provisioning_instance_details` but no `gateway_subdomain`/`gateway_ssh_port` yet (and no connectable direct IP), the user sees: *"Gateway routing is being configured. **User impact:** Tenant's VM just provisioned; they're staring at the connection-details card; "shortly" gives them no signal whether to wait 30 seconds or open a support ticket; they refresh the whole page repeatedly.
**Fix sketch:** Add a small inline "Refresh connection status" button scoped to this contract, plus an estimated wait ("typically 1–3 minutes after provisioning completes"). Optionally add a `waitForSelector`-style auto-poll.

### 11. Failed-email table truncates the error message
**Route + lines:** `/dashboard/admin/+page.svelte:941-942`
**Severity:** P1
**Fix size:** small

```svelte
<td class="py-3 px-2 text-xs text-red-400 max-w-xs truncate">
  {email.lastError || "Unknown error"}
</td>
```

The admin's primary purpose for opening this table is to read why emails are failing. `max-w-xs truncate` cuts the SMTP/transport error to one line. The full error is not exposed in any tooltip, expandable row, or details view.

**User impact:** Admin sees "Email relay timeout while connecting to…" truncated mid-sentence; cannot diagnose; has to dig through server logs to find the full error.
**Fix sketch:** Remove `truncate`, wrap the cell in a `<details>`/`<summary>` with the full error inside a `<pre>`, or add a `title={email.lastError}` attribute as a stopgap.

### 12. ~~WelcomeModal: backdrop click permanently dismisses onboarding~~ **FIXED**
**Resolution (2026-07-21, commit `5fff8ea9`):** Split `finishOnboarding()` into a separate `closeModal()` that only sets `open=false`; backdrop now calls `closeModal` while `finishOnboarding` (which writes the `first_login_onboarding_completed` flag) is reserved for the three explicit CTAs. RED→GREEN test in `first-login-onboarding.spec.ts` asserts the flag is NOT set after a backdrop click (using `page.evaluate` because the fast-auth fixture's context-level init script writes 'true' on every navigation, masking reload-based behavior).
**Route + lines:** `/lib/components/WelcomeModal.svelte:71` and `welcome-onboarding.ts:12-14`
**Severity:** P1
**Fix size:** small

The backdrop `<div class="absolute inset-0 ..." onclick={finishOnboarding} role="presentation">` calls `finishOnboarding` on any click. `finishOnboarding` then sets `sessionStorage['first_login_onboarding_completed'] = 'true'` and closes the modal. A new user who clicks anywhere outside the modal during step 1 (a common reflex to "see what's behind this") silently skips all three onboarding steps with no way back.

**User impact:** First-time registrant clicks the dim background to peek at the dashboard; expects to bring the modal back; onboarding never reappears; they never get the SSH-key prompt.
**Fix sketch:** Decouple "close" from "complete": backdrop click should set `open=false` *without* calling `completeOnboarding`, so the modal reappears on next visit to `/dashboard`. Only the explicit "Continue" path through step 3 (or the explicit "Stay on dashboard" button) should call `completeOnboarding`. Test by simulating a backdrop click and reloading.

### 14. Rentals "failed" / "rejected" states have no next-step CTA
**Route + lines:** `/dashboard/rentals/+page.svelte:116-148` (`getNextStepInfo`)
**Severity:** P2
**Fix size:** small

`getNextStepInfo` has explicit branches for `requested`/`pending`/`accepted`/`provisioning`/`provisioned`/`active`/`rejected`/`cancelled`, but **not** for `failed`. A `failed` contract returns `null`, so the lifecycle progress card just shows the status badge with no next-step text. Even the `rejected` branch tells the user "You can try another provider" without linking to the marketplace.
**User impact:** Tenant's contract fails provisioning; expects guidance; sees only the "Failed" badge; must figure out on their own that the marketplace link in the sidebar is the next step.
**Fix sketch:** Add a `failed` branch returning something like *"Provisioning failed. Contact the provider or browse the marketplace for alternatives"* with a link. Add a marketplace link to the `rejected` branch's text. Single-function edit.

### 15. `OfferingStatusBadge` tooltip is hover-only (keyboard-inaccessible)
**Route + lines:** `/lib/components/OfferingStatusBadge.svelte:103-114`
**Severity:** P2 (a11y)
**Fix size:** small

The "more details" info button uses `onmouseenter`/`onmouseleave` to show/hide the tooltip, plus an `onclick` toggle. There are no `onfocus`/`onblur` handlers, so a keyboard-only user navigating with Tab cannot reach the tooltip content (Trust score, Subscription, Has setup recipe, Has warnings). The tooltip `role="tooltip"` is set but isn't associated with the trigger via `aria-describedby`.
**Fix sketch:** Add `onfocus`/`onblur` mirroring the hover handlers, add `aria-describedby` pointing at the tooltip id, and add Escape-to-close. Small fix.
